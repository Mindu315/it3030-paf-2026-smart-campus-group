# Resource Management & Resource Catalog Backend README

## Overview
This backend module provides Resource Management and Resource Catalog APIs for Smart Campus.
It supports create, read, update, delete, and filtering of resources stored in MongoDB.

Base URL:
- `http://localhost:8080`

Main API base:
- `/api/resources`

## Tech Stack
- Spring Boot `3.4.0`
- Java `21`
- MongoDB (Atlas URI configured)
- Spring Data MongoDB
- Spring Web
- Spring Security (custom route permissions)

## Current Configuration
From `src/main/resources/application.properties`:

```properties
server.port=8080
spring.data.mongodb.uri=mongodb+srv://db:1234@cluster0.qe5skxx.mongodb.net/Smart_Campus_DB?retryWrites=true&w=majority
spring.mongodb.uri=mongodb+srv://db:1234@cluster0.qe5skxx.mongodb.net/Smart_Campus_DB?retryWrites=true&w=majority
```

## Security & CORS
- These routes are publicly accessible:
- `/`
- `/api/resources/**`
- `/api/debug/**`
- `/api/users/register`
- `/api/users/login`
- Other routes require authentication.

CORS allowed origins:
- `http://localhost:3000`
- `http://localhost:3001`
- `http://localhost:5173`

Allowed methods:
- `GET`, `POST`, `PUT`, `DELETE`, `OPTIONS`

## Data Model
Mongo collection:
- `resources`

Resource fields:
- `id` (String, generated by MongoDB)
- `name` (String, required, not blank)
- `type` (String, required, not blank)
- `capacity` (Integer, required, minimum 1)
- `location` (String, required, not blank)
- `availabilityStart` (String, required, not blank)
- `availabilityEnd` (String, required, not blank)
- `status` (String, required, not blank)
- `description` (String, optional)

## DTOs

Response DTO (`ResourceResponseDto`):
- `id`

- `type`
- `capacity`

- `availabilityStart`
- `availabilityEnd`

## API Endpoints

### 1. Health/Home
Method:
- `GET /`

Response example:
```json
{
  "message": "Smart Campus Resource Service is running",
  "resourcesEndpoint": "/api/resources"
}
```

### 2. Create Resource
Method:
- `POST /api/resources`

Request body:
```json
{
  "name": "Lab A",
  "type": "LAB",
  "capacity": 40,
  "location": "Building B - Floor 2",
  "availabilityStart": "08:00",
  "availabilityEnd": "18:00",
  "status": "AVAILABLE",
  "description": "Computer lab with 40 seats"
}
```

Success response:
- Status `201 Created`
- Body: created `ResourceResponseDto`

### 3. Get All Resources
Method:
- `GET /api/resources`

Success response:
- Status `200 OK`
- Body: array of `ResourceResponseDto`

### 4. Get Resource By ID
Method:
- `GET /api/resources/{id}`

Success response:
- Status `200 OK`
- Body: `ResourceResponseDto`

Not found response:
- Status `404 Not Found`
```json
{
  "message": "Resource not found with id: <id>"
}
```

### 5. Update Resource
Method:
- `PUT /api/resources/{id}`

Request body:
- Same format as Create Resource

Success response:
- Status `200 OK`
- Body: updated `ResourceResponseDto`

### 6. Delete Resource
Method:
- `DELETE /api/resources/{id}`

Success response:
- Status `204 No Content`

### 7. Filter Resources
Method:
- `GET /api/resources/filter`

Optional query params:
- `type` (String)
- `location` (String)
- `minCapacity` (Integer)
- `status` (String)

Example:
- `GET /api/resources/filter?type=LAB&location=Building%20B%20-%20Floor%202&minCapacity=30&status=AVAILABLE`

Success response:
- Status `200 OK`
- Body: array of `ResourceResponseDto` matching all provided filters

## Debug Endpoint
Method:
- `GET /api/debug/mongo`

Purpose:
- Verify active Mongo URI, DB, collection, and current resource count

Example response:
```json
{
  "mongoUri": "mongodb+srv://...",
  "database": "Smart_Campus_DB",
  "collection": "resources",
  "resourceCount": 1
}
```

## Error Handling Format

Validation errors (`400 Bad Request`):
```json
{
  "name": "Name is required",
  "capacity": "Capacity must be at least 1"
}
```

Malformed JSON/body type errors (`400 Bad Request`):
```json
{
  "message": "Invalid request body. Please send valid JSON with correct field types."
}
```

Database errors (`500 Internal Server Error`):
```json
{
  "message": "Database operation failed",
  "error": "<database driver message>"
}
```

Generic server errors (`500 Internal Server Error`):
```json
{
  "message": "An unexpected error occurred",
  "error": "<exception message>"
}
```

## Frontend Integration Notes
- Use `http://localhost:8080` as API host.
- Resource APIs are currently open (no token required) per security config.
- `availabilityStart` and `availabilityEnd` are plain strings. Frontend should enforce a consistent format like `HH:mm`.
- `status` and `type` are free strings in backend. Frontend should use controlled dropdown values for consistency.
- For update/create, send all required fields.
- Handle `404` for deleted/non-existing resource IDs.
- Handle `400` validation object responses by mapping field keys to form error messages.

## Frontend Setup (Project Integration)

### 1. Install and run frontend
From project root:

```bash
cd frontend
npm install
npm run dev
```

Default frontend URL:
- `http://localhost:5173`

### 2. Ensure frontend API base URL uses backend port 8080
Current frontend API client file:
- `frontend/src/api/axiosConfig.js`

Set base URL to:

```js
import axios from "axios";

const api = axios.create({
  baseURL: "http://localhost:8080/api",
});

export default api;
```

### 3. Run backend before testing resource screens
In another terminal:

```bash
cd backend
mvn spring-boot:run
```

Backend URL:
- `http://localhost:8080`

### 4. Frontend calls for Resource Management
- Get all resources: `GET /api/resources`
- Get one resource: `GET /api/resources/{id}`
- Create resource: `POST /api/resources`
- Update resource: `PUT /api/resources/{id}`
- Delete resource: `DELETE /api/resources/{id}`
- Filter catalog: `GET /api/resources/filter?type=&location=&minCapacity=&status=`

### 5. Quick frontend test flow
- Open frontend app (`5173`)
- Navigate to Resource Catalog page
- Create one resource and confirm it appears in list
- Apply filter by `type` or `status`
- Edit the same resource
- Delete it and confirm it is removed
